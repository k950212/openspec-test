## Context

目前 storefront 的商品列表卡片已提供查看商品、收藏與加入購物車等操作，但使用者若只想先快速確認商品資訊，仍需進入商品詳情頁。這次 change 想補上一個不離開列表頁的 quick view 體驗，因此設計重點會落在 `HomeView.vue` 內以最小範圍延伸既有商品卡操作、維持現有篩選與分頁狀態、並讓 quick view 與既有 modal/overlay 互動風格保持一致。

## Goals / Non-Goals

**Goals:**
- 在商品列表中提供可快速開啟的商品預覽入口。
- 讓 quick view 顯示足夠的核心資訊，包含圖片、名稱、價格、描述與主要操作。
- 保持與既有收藏、加入購物車、查看商品頁流程一致，不重複實作另一套商品邏輯。
- 讓 quick view 在桌機與小螢幕上都可關閉、可閱讀、可操作。

**Non-Goals:**
- 不建立新的路由頁面或後端 API。
- 不在 quick view 中完整複製商品詳情頁的所有模組，例如評論、相關商品或完整變體選擇流程。
- 不重做整個商品列表卡片版型，只增加必要入口與預覽容器。

## Decisions

### 1. 在 `HomeView.vue` 內以本地 state 管理 quick view 開關與目前商品
- 決策：使用本地 `ref` 管理目前預覽商品與 dialog 開啟狀態，而不是新增全域 store。
- 理由：quick view 的生命週期完全屬於商品列表頁，使用本地 state 可降低耦合並避免把暫時 UI 狀態擴散到全域。
- 替代方案：把 quick view 狀態放進 Pinia store。這會增加共享狀態複雜度，但目前沒有跨頁共享需求。

### 2. Quick view 只顯示商品核心資訊與導向操作
- 決策：dialog 內顯示商品主圖、名稱、價格、分類、簡短描述，以及「查看商品」與可行時的「加入購物車」操作；若商品有 variants，沿用既有提示並導向商品頁選規格。
- 理由：這能滿足快速預覽目的，同時避免在 modal 中複製完整商品詳情頁與變體處理。
- 替代方案：在 quick view 中完整支援變體選擇。這會拉高實作複雜度，也容易與既有商品詳情頁邏輯重疊。

### 3. 使用 overlay dialog 並保留明確關閉行為
- 決策：採用列表頁內的 modal/dialog overlay，支援背景遮罩點擊關閉、按鈕關閉，以及在小螢幕上維持可捲動內容。
- 理由：quick view 是暫時性預覽，使用 overlay 最符合使用者心智模型，也能保留當前列表的篩選、排序與分頁上下文。
- 替代方案：使用 drawer。雖然也可行，但目前卡片資訊與預覽內容量較適合中央 dialog，且與現有圖片預覽 modal 的互動更接近。

### 4. Quick view 不重置 catalog 篩選與分頁狀態
- 決策：開關 quick view 不影響現有 catalog 的搜尋、價格篩選、排序、分頁與 loading/empty state。
- 理由：使用者常在多個商品間連續比較，若 modal 操作造成列表上下文改變，會讓瀏覽體驗斷裂。
- 替代方案：關閉 quick view 後重新整理列表區塊。這會增加不必要的跳動與心智負擔。

## Risks / Trade-offs

- [Risk] Quick view 與商品卡既有按鈕並存後，操作密度變高。 → Mitigation：將 quick view 視為次要但清楚的入口，避免壓縮主要 CTA 可點擊面積。
- [Risk] 在 modal 中再次觸發加入購物車或前往商品頁，可能與既有商品卡流程出現重複文案或狀態同步問題。 → Mitigation：直接重用既有加入購物車與導頁邏輯，不建立分叉版本。
- [Risk] 小螢幕上的 dialog 若內容過高，可能造成操作區被擠壓。 → Mitigation：限制容器寬高、允許內容捲動，並保留固定可見的關閉入口。
