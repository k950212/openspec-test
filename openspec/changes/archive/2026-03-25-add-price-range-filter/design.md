## Context

目前 storefront 首頁已支援商品搜尋、分類切換、排序與 empty state，但還缺少價格區間篩選。商品資料目前由前端靜態資料提供，價格欄位已存在，因此這次變更可以直接在前端篩選流程中擴充，不需要後端 API 或資料模型遷移。

## Goals / Non-Goals

**Goals:**
- 在商品列表頁提供最低價與最高價篩選。
- 讓價格區間與現有搜尋、分類、排序共同作用，輸出單一一致的結果集。
- 在輸入無效區間或沒有結果時，提供清楚的回饋與重設方式。

**Non-Goals:**
- 不新增後端查詢、伺服器端分頁或資料庫篩選。
- 不處理多幣別、促銷價、會員價等進階價格模型。
- 不新增 histogram、slider 視覺化價格軸等進階篩選元件。

## Decisions

### 使用最低價與最高價的雙欄位輸入
以兩個明確欄位表示價格下限與上限，能最直接對應目前簡單的 catalog 篩選需求，也比較容易和現有版面整合。相較 range slider，雙欄位在桌機與手機都更容易控制，實作與驗證也較單純。

### 將價格區間整合進既有前端 computed 篩選管線
商品資料已經在前端記憶體中，因此最穩定的做法是把價格篩選放進目前的商品列表計算流程，依序與搜尋、分類、排序一起處理。這可以避免多份結果狀態不同步，也能讓 empty state 只依最後結果集判斷。

### 無效區間採用即時阻擋與可重設策略
當使用者輸入的最低價大於最高價時，頁面應顯示明確提示，並避免把這個條件當作有效篩選套用。這比自動交換數值更可預期，也能避免使用者不理解系統偷偷改值。畫面同時提供清除價格範圍的入口，讓使用者快速回復完整列表。

## Risks / Trade-offs

- [Risk] 價格區間和既有搜尋、分類同時套用後，使用者可能不清楚是哪個條件造成沒有結果 → Mitigation：在 empty state 明確指出目前有篩選條件，並提供清除價格區間的操作。
- [Risk] 手機版同時擺放兩個價格輸入欄位可能壓縮版面 → Mitigation：沿用現有篩選區排版，必要時改為直向堆疊，優先維持可輸入性與可讀性。
- [Risk] 未來若加入折扣價或多幣別，現行邏輯需要再抽象化 → Mitigation：先以商品現有單一 `price` 欄位為唯一篩選來源，避免過早設計。
