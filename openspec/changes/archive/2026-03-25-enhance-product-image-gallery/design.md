## Context

目前商品頁只有基礎的單張圖片呈現，無法提供更貼近電商網站的圖片瀏覽體驗。你更新後的 proposal 已經把範圍擴大到多張縮圖、主圖切換、放大預覽 modal，以及手機上的 swipe 瀏覽，因此這次設計需要在現有商品頁結構中，把圖片區塊升級為完整但仍可漸進降級的互動式 gallery。

現有商品資料主要依賴單一 `image` 欄位，商品頁同時還承載 variants、wishlist、cart、reviews 與 related products。這代表圖片畫廊必須盡量局部實作，避免干擾既有商品詳情流程。

## Goals / Non-Goals

**Goals:**
- 保留現有 `image` 欄位相容性，同時支援商品定義多張 gallery 圖片。
- 在商品頁提供主圖與縮圖導覽，並支援桌機 hover / click 與手機 click / swipe 的切換方式。
- 提供可放大檢視主圖的 preview modal，讓使用者能更仔細查看商品圖片。
- 在切換商品或切換會影響圖片集的 variant 時，正確重設目前選取圖片。
- 讓單圖商品維持簡潔呈現，不出現多餘控制元件。

**Non-Goals:**
- 不新增後台圖片上傳、裁切或管理介面。
- 不引入第三方輪播或手勢套件，優先以現有前端能力完成。
- 不重新設計商品頁其他功能區塊，如評論、推薦商品或購物車邏輯。

## Decisions

### 1. 商品資料使用「主圖 + gallery 圖片列表」結構
- 決策：保留現有 `image` 作為主圖來源，另外增加 `gallery?: string[]` 以支援額外圖片。
- 理由：這樣可以維持首頁、推薦商品與其他現有畫面繼續使用單張圖欄位，不需要整個 catalog 一起重構。
- 替代方案：全面改成 `images: string[]`。這會讓既有依賴 `image` 的畫面都必須同步改寫，範圍過大。

### 2. 商品頁以本地 state 管理目前選取圖片與 modal 狀態
- 決策：在 `ProductDetailView.vue` 內管理 active image index、preview modal 開關，以及 swipe 的互動狀態。
- 理由：這些互動都只屬於單一商品頁，不值得提升到 store；留在本地 state 也更容易在切換商品時重設。
- 替代方案：把 gallery 狀態放進 store 或 route query。這會增加狀態同步成本，對目前需求沒有明顯收益。

### 3. 互動方式依裝置情境漸進增強
- 決策：縮圖在桌機支援 hover 與 click 切換，在觸控裝置則以 click/tap 為主；主圖區在手機版支援 swipe 瀏覽。
- 理由：proposal 明確要求不同裝置互動模式，而這種設計能兼顧可用性與實作複雜度。
- 替代方案：所有裝置統一只用 click。雖然較簡單，但會少掉桌機常見的快速 hover 預覽體驗。

### 4. Preview modal 與圖片集切換要支援降級與容錯
- 決策：modal 只在使用者主動開啟時顯示，並使用與主圖相同的 gallery source；若圖片載入失敗，保留穩定版面與可繼續切換的能力。
- 理由：proposal 的 acceptance criteria 與 edge cases 都要求放大預覽與圖片失敗情境，因此需要把 modal 視為 gallery 的延伸，而不是另一套獨立資料來源。
- 替代方案：只做主圖放大、不支援 modal 內切換。這會讓體驗斷裂，也不利於處理多圖瀏覽。

## Risks / Trade-offs

- [圖片互動變多，商品頁複雜度上升] → 將 gallery 狀態限制在圖片區塊，避免影響其他商品頁功能。
- [手機 swipe 與桌機 hover 行為不同] → 使用清楚的縮圖選取狀態與保守的手勢判定，降低誤觸。
- [單圖商品仍被迫顯示多圖 UI] → 在只有一張圖時隱藏縮圖列與非必要提示，維持簡潔版面。
- [variant 之後若帶入不同圖片集] → 設計上預留圖片集重算與 active image reset 流程，避免之後再重構一次。
- [某張圖片載入失敗] → 讓主圖與縮圖容器維持固定尺寸與後備視覺狀態，避免版面跳動。
