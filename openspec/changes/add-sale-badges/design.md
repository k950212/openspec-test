## Context

目前 storefront 的商品卡已經承載商品圖片、名稱、價格、快速預覽、收藏與加入購物車等資訊，但缺少能快速傳達促銷訊息的視覺標示。這次 change 會在現有產品資料與商品卡版型上增加 sale badge 支援，因此設計重點是用最小資料模型擴充、讓 badge 能跨多個卡片表面共用，並確保文字可讀性不會被商品圖片或按鈕區干擾。

## Goals / Non-Goals

**Goals:**
- 讓商品資料可宣告前端用的 sale badge 文案與樣式。
- 在商品列表與其他共用商品卡介面穩定顯示 sale badge。
- 確保 badge 在不同圖片背景、小螢幕尺寸與既有卡片按鈕共存時仍然清楚可讀。
- 保持功能為純前端展示，不影響價格計算與結帳邏輯。

**Non-Goals:**
- 不引入自動折扣、原價/特價雙價格、優惠倒數或結帳折抵計算。
- 不串接後端促銷 API 或會員分級定價系統。
- 不建立複雜的活動規則引擎，只支援靜態 badge 顯示。

## Decisions

### 1. 在 `src/data/products.ts` 內新增輕量 sale badge 欄位
- 決策：以商品資料欄位儲存 badge 文案與視覺 variant，而不是從價格或 coupon 資訊推導。
- 理由：目前專案仍是 frontend-only，使用顯式欄位最直觀，也能避免把展示 badge 與價格邏輯耦合。
- 替代方案：依價格門檻或 coupon 規則自動推算 badge。這會模糊 badge 的意圖，也不利於內容控制。

### 2. 以共用樣式模式渲染 sale badge
- 決策：在商品卡圖片區塊上方渲染 badge，並用一致的 class 與 variant 樣式控制外觀。
- 理由：badge 放在圖片區最容易被快速掃視，也不會擠壓標題、價格與操作區。
- 替代方案：將 badge 放在標題或價格附近。這會增加文字區密度，對卡片高度更敏感。

### 3. 先覆蓋主要商品卡表面，再保持擴充空間
- 決策：優先支援首頁商品列表、quick view 相關呈現，以及其他使用相同卡片模式的地方；若某些卡片沒有圖片覆蓋區，再使用次要 badge 位置。
- 理由：這能在最常被瀏覽的表面上建立一致心智模型，同時保留後續延伸空間。
- 替代方案：只在首頁顯示 badge。這會讓使用者在其他卡片表面失去一致預期。

### 4. Badge 只負責促銷標示，不改變既有操作流程
- 決策：sale badge 僅作為視覺提示，不影響 quick view、wishlist、cart、search、filter 或 pagination 邏輯。
- 理由：proposal 明確將範圍限定在展示層，這樣風險最低，也避免造成規格外的互動副作用。
- 替代方案：點擊 badge 打開活動說明或自動套用促銷。這超出本次需求。

## Risks / Trade-offs

- [Risk] Badge 疊在圖片上時，某些商品圖背景可能讓文字不夠清楚。 → Mitigation：使用高對比底色、圓角膠囊樣式與陰影，必要時加半透明背景。
- [Risk] 多個卡片表面共用 badge 後，樣式差異可能導致視覺不一致。 → Mitigation：建立統一 badge variant 命名與基礎樣式，局部只調整定位。
- [Risk] 未來若加入真實促銷價或倒數活動，現有靜態 badge 模型可能不足。 → Mitigation：先將 badge 視為獨立展示層欄位，後續仍可擴充資料結構而不影響目前卡片佈局。
